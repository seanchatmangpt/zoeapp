## Proof Cell Verification

This checkpoint establishes the "Proof Cell Verification" protocol for the ZoeApp research program. The objective is to maintain a high-integrity, secure-by-default architecture in the Ostar generative pipeline.

### Verification Rules

1.  **Reject Client-Only Trust:** All database operations must be mediated by defined RLS policies. Direct exposure of tables to `anon` or `authenticated` roles without strict RLS constraints is prohibited.
2.  **Reject Service-Role Exposure:** Use of `service_role` in client-side code, frontend applications, or unauthorized edge functions is strictly forbidden. Any operations requiring elevated privileges must be handled through authorized, hardened backend endpoints.
3.  **Verify RLS Coverage:** Every table in the `public` schema must have comprehensive RLS policies defining `SELECT`, `INSERT`, `UPDATE`, and `DELETE` access.

### Audit Mechanism

The Proof Cell Verification protocol utilizes the `supabase db advisors` suite to perform automated security scans. These scans are integrated into the CI/CD pipeline and must pass for any code merge.

-   **Audit Script:** `zoeapp-research-program/audits/audit_security.sh`
-   **Enforcement:** The audit script will exit with a non-zero status if any security vulnerabilities of `warn` or `error` level are detected.
