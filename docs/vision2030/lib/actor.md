# Actor Subsystem: Supabase Realtime-Powered BEAM Actor Runtime

The `actor` library provides a BEAM-style concurrent Actor runtime built on top of Supabase Realtime and local sqlite database persistence. It enables local optimistic transaction simulation, automated state rollback upon remote authority rejection, fault supervision boundaries, and decentralized replication of virtual knowledge graph states.

---

## 1. Title & Overview

The Actor Subsystem (located in [src/lib/actor](file:///Users/sac/zoeapp/src/lib/actor)) is a low-level module designed to support transactional, distributed, and deterministic state transitions across the Truex platform. By adopting actor semantics modeled after the Erlang/BEAM architecture, each entity in the system is represented as a stateful actor identified by its tenant, kind, and unique ID.

Key capabilities of the actor library include:
- **Optimistic Local Simulation**: Mutates local virtual knowledge graphs instantly to maintain a fluid user experience even during network partitions or offline states.
- **Outbox-Driven Reconciliation**: Queues commands for execution on remote authoritative systems and performs asynchronous synchronization.
- **Compensating Rollback**: Reverts local semantic database mutations (quads) if the remote authority rejects the simulated transaction.
- **Fault-Tolerant Supervision**: Implements robust retry policies with exponential backoff under isolation boundaries to guard against transient computation failures.
- **Replay & Divergence Auditing**: Compares the expected outcomes of replayed commands with actual recorded database events to identify discrepancies, unauthorized role executions, or broken causal sequences.

## 2. Architectural & Philosophical Mapping

The Actor library is a core implementation of the Truex architectural blueprint and maps directly to its semantic and security layers.

### Architectural Mapping
- **Membrane**: Governs mutation rights and manages system boundaries. As demonstrated in [actorOps.ts](file:///Users/sac/zoeapp/src/lib/actor/actorOps.ts), dev/simulator mutations (e.g., changes to network state or current principal) are wrapped inside a proxy-based `ProxyableBridge` governed by a strict `MembraneContext`. Any mutation executes through traps that emit `MembraneReceipts` for strict security and execution verification.
- **Intake**: Intake operations are structured via `CommandEnvelope` payloads. Command validators inspect incoming parameters, check signatures, and enforce authorization rules prior to execution.
- **Projection**: The state transitions generated by commands are represented as RDF `ConstructDelta` payloads (a set of `add` and `remove` RDF `Quad` operations). When applied, these modify the local `VirtualKnowledgeGraphClient`, directly updating the semantic data model of the app.
- **Supervision**: Isolation of runtime crashes is managed by the `ActorSupervisor`. The supervisor executes state construction and validation inside try-catch fault boundaries, applying exponential backoff retry algorithms to isolate memory and execution spaces.

### Philosophical Mapping: The Chatman Equation

The behavior of the Actor system is mathematically modeled by the Chatman Equation:

$$R \vdash A = \mu(O^*)$$

Where:
- **$R$ (Role / Authority)**: Represents the execution context, defined by the actor's tenant context, system settings, and the executing user's `Principal` role (`admin`, `pastor`, `member`, `guest`).
- **$A$ (Action / Command)**: Represents the `CommandEnvelope` containing the command string, parameters, and unique identifier.
- **$\mu(O^*)$ (Mutation / State Space Transformation)**: The compiled `ConstructDelta` containing the exact set of RDF `Quad` operations to add or remove from the virtual knowledge graph.
- **$\vdash$ (Entails / Validation Gate)**: The validation check performed by the dispatcher. If the Principal's role $R$ and payload validation satisfy the specifications defined in the registered `CommandSpec`, the gate resolves to true, and the mutation $\mu(O^*)$ is committed to the graph.

## 3. Source Code Structure

The library consists of 6 primary source files located in [src/lib/actor](file:///Users/sac/zoeapp/src/lib/actor):

1. **[types.ts](file:///Users/sac/zoeapp/src/lib/actor/types.ts)**  
   Defines the core data contracts, including `ActorRef`, `Principal`, `CommandEnvelope`, `Receipt`, `EventEnvelope`, `ConstructDelta`, `CommandSpec`, `QuerySpec`, `SupervisionPolicy`, and behavior contracts.

2. **[registry.ts](file:///Users/sac/zoeapp/src/lib/actor/registry.ts)**  
   Implements a thread-safe singleton registry (`ActorRegistry`) mapping `actorKind` identifiers to behavior specifications, validator functions, query selectors, and supervisor configurations.

3. **[supervision.ts](file:///Users/sac/zoeapp/src/lib/actor/supervision.ts)**  
   Contains the `ActorSupervisor` which wraps state evaluation loops in protective boundaries, preventing infinite loops or crash cascades using customizable retry counts and exponential backoffs.

4. **[replayDiff.ts](file:///Users/sac/zoeapp/src/lib/actor/replayDiff.ts)**  
   Provides audit tools (`analyzeSingleCommandReplay`) to detect state drift between simulated and actual states, highlighting anomalies like missing transitions, unexpected mutations (hash mismatches), role authorization discrepancies, or broken causal sequences.

5. **[dispatcher.ts](file:///Users/sac/zoeapp/src/lib/actor/dispatcher.ts)**  
   The core execution coordinator (`ActorDispatcher`). It manages local simulation, inserts outbox requests, executes remote authoritative transactions, syncs logs to Supabase via `ActorSyncEngine`, and processes rollback operations upon server-side transaction rejection.

6. **[actorOps.ts](file:///Users/sac/zoeapp/src/lib/actor/actorOps.ts)**  
   Acts as the operational cockpit, storing network configurations, the active principal, drop rates, and latest receipt state inside a `zustand` store. It wraps runtime variables under a security membrane.

## 4. API Contracts

### Data Types & Interfaces ([types.ts](file:///Users/sac/zoeapp/src/lib/actor/types.ts))

```typescript
export interface ActorRef {
  tenantId: string;
  kind: string;
  id: string;
}

export interface Principal {
  id: string;
  role: 'admin' | 'pastor' | 'member' | 'guest';
}

export interface CommandEnvelope<T = any> {
  id: string;
  actor: ActorRef;
  command: string;
  principal: Principal;
  payload: T;
  idempotencyKey: string;
  causationId?: string;
  correlationId?: string;
}

export type ReceiptStatus =
  | 'accepted_pending'
  | 'rejected_local'
  | 'applied_local'
  | 'applied_remote'
  | 'rejected_remote'
  | 'quarantined';

export interface Receipt {
  id: string;
  commandId: string;
  actor: ActorRef;
  status: ReceiptStatus;
  deltaHash?: string;
  eventIds: string[];
  createdAt: string;
  error?: string;
}

export interface EventEnvelope<T = any> {
  id: string;
  commandId: string;
  actor: ActorRef;
  type: string;
  payload: T;
  createdAt: string;
}

export interface ConstructDelta {
  add: Quad[];
  remove: Quad[];
}

export interface CommandSpec<TInput = any> {
  roles: ('admin' | 'pastor' | 'member' | 'guest')[];
  inputValidator: (payload: any) => boolean | Promise<boolean>;
  construct: (payload: TInput, actor: ActorRef) => ConstructDelta | Promise<ConstructDelta>;
  emits: string[];
}

export interface QuerySpec<TInput = any, TOutput = any> {
  select: (params: TInput, store: any) => TOutput | Promise<TOutput>;
}

export interface SupervisionPolicy {
  maxRetries: number;
  backoffMs: number;
  strategy: 'restart' | 'resume' | 'stop';
}

export interface ActorBehavior {
  actorKind: string;
  commands: Record<string, CommandSpec>;
  queries: Record<string, QuerySpec>;
  supervise?: SupervisionPolicy;
}

export type DispatchMode = 'local' | 'remote';

export interface DispatchContext {
  mode: DispatchMode;
  authority: 'optimistic' | 'authoritative';
}
```

### ActorRegistry ([registry.ts](file:///Users/sac/zoeapp/src/lib/actor/registry.ts))
- `static getInstance(): ActorRegistry`  
  Returns the registry singleton instance.
- `register(behavior: ActorBehavior): void`  
  Registers a behavior spec for a specific `actorKind`. Throws an error if the kind is already registered.
- `resolve(kind: string): ActorBehavior`  
  Retrieves behavior for an actor kind. Throws if missing.
- `clear(): void`  
  Clears all registered behaviors.

### ActorSupervisor ([supervision.ts](file:///Users/sac/zoeapp/src/lib/actor/supervision.ts))
- `static async execute<T>(operation: () => Promise<T>, policy?: SupervisionPolicy): Promise<T>`  
  Runs the specified closure using the designated supervision guidelines. Automatically performs exponential backoff retries when computation crashes.

### Replay Differential Analyzer ([replayDiff.ts](file:///Users/sac/zoeapp/src/lib/actor/replayDiff.ts))
- `analyzeSingleCommandReplay(envelope: CommandEnvelope, observedReceipt: Receipt | null, expectedDeltaHash: string, allowedRoles: string[], replayedSuccess: boolean, replayedError?: string): ReplayDivergence[]`  
  Performs structural reconciliation on five distinct categories of state discrepancies:
  1. **missing transition**: No matching execution receipt found for the command.
  2. **unexpected mutation**: Observed delta hash mismatches expected delta hash.
  3. **authority mismatch**: Executing role lacks privilege based on rules.
  4. **receipt inconsistency**: Contradiction between observed status (success vs fail) or differing error strings.
  5. **causal divergence**: Disruption in correlation/causation ID patterns.

### ActorDispatcher ([dispatcher.ts](file:///Users/sac/zoeapp/src/lib/actor/dispatcher.ts))
- `constructor(vkgClient: VirtualKnowledgeGraphClient, context?: DispatchContext, syncEngine?: ActorSyncEngine)`  
  Initializes the dispatcher with a target graph client, dispatching context, and optional sync engine.
- `dispatch<TPayload>(envelope: CommandEnvelope<TPayload>): Promise<Receipt>`  
  Dispatches the command. Under `'local'` mode, records a local `'accepted_pending'` receipt, applies optimistic changes, and inserts an outbox task. Under `'remote'` mode, compiles and executes authority validation directly.
- `syncOutbox(remoteDispatcher: ActorDispatcher): Promise<void>`  
  Scans local outbox jobs and pushes pending items to the remote authority. If rejected, it extracts optimistic quads from the payload and rolls them back by performing opposite operations (removing added quads and re-adding removed quads), marking records as `'failed'` and recording to quarantine logs.

### Database Schemas ([src/lib/db/schema.ts](file:///Users/sac/zoeapp/src/lib/db/schema.ts))
The subsystem persists state using SQLite tables managed by Drizzle ORM:
- **`actor_commands`**: Tracks incoming commands, causation/correlation IDs, payloads, and processing statuses (`pending`, `processing`, `applied`, `rejected`).
- **`actor_events`**: Stores domain events emitted during execution.
- **`actor_receipts`**: Records definitive outcomes, hashes of changes, and execution errors.
- **`actor_outbox`**: Holds outbound requests to be dispatched to the server when network access is online.
- **`actor_quarantine`**: Contains rejected payloads alongside their authoritative errors for diagnosis.

## 5. Usage Guide

Below is a complete, production-ready implementation showcasing actor behavior configuration, local optimistic updates, outbox synchronization, error handling, and rollback execution.

```typescript
import { ActorDispatcher, ActorSyncEngine } from '../src/lib/actor/dispatcher';
import { ActorRegistry } from '../src/lib/actor/registry';
import { VirtualKnowledgeGraphClient } from '../src/lib/vkg/client';
import { DataFactory } from '../src/lib/vkg/rdf';
import { ActorRef, CommandEnvelope, ActorBehavior } from '../src/lib/actor/types';
import { 
  setNetworkOffline, 
  setCurrentPrincipal, 
  useActorOpsStore 
} from '../src/lib/actor/actorOps';

async function runActorWorkflow() {
  // 1. Initialize Virtual Knowledge Graph client
  const vkgClient = new VirtualKnowledgeGraphClient();

  // 2. Define behavior for a 'MemberProfile' Actor Kind
  const memberProfileBehavior: ActorBehavior = {
    actorKind: 'MemberProfile',
    supervise: {
      maxRetries: 3,
      backoffMs: 20,
      strategy: 'restart',
    },
    commands: {
      UpdateDisplayName: {
        roles: ['admin', 'pastor', 'member'],
        inputValidator: async (payload: any) => {
          return typeof payload.name === 'string' && payload.name.trim().length > 2;
        },
        construct: async (payload: { name: string }, actor: ActorRef) => {
          const subject = DataFactory.namedNode(`https://truex.org/member/${actor.id}`);
          const predicate = DataFactory.namedNode('https://schema.org/name');
          const newObject = DataFactory.literal(payload.name);

          // We construct a delta replacing any old name
          return {
            add: [DataFactory.quad(subject, predicate, newObject, DataFactory.defaultGraph())],
            remove: [], // If we had previous values, we would query and list them here
          };
        },
        emits: ['DisplayNameUpdated'],
      },
    },
    queries: {},
  };

  // 3. Register behavior
  const registry = ActorRegistry.getInstance();
  registry.clear();
  registry.register(memberProfileBehavior);

  // 4. Instantiate local and remote dispatchers
  const syncEngine = new ActorSyncEngine();
  const localDispatcher = new ActorDispatcher(vkgClient, { mode: 'local', authority: 'optimistic' }, syncEngine);
  const remoteDispatcher = new ActorDispatcher(vkgClient, { mode: 'remote', authority: 'authoritative' });

  // 5. Construct a Command Envelope for the update operation
  const envelope: CommandEnvelope<{ name: string }> = {
    id: `cmd_${Date.now()}`,
    actor: {
      tenantId: 'tenant-zoe',
      kind: 'MemberProfile',
      id: 'profile-99',
    },
    command: 'UpdateDisplayName',
    principal: {
      id: 'usr-member-99',
      role: 'member',
    },
    payload: { name: 'Elijah Stone' },
    idempotencyKey: `idem_${Date.now()}`,
  };

  console.log('--- Step 1: Executing local optimistic simulation ---');
  // Dispatch local optimistic simulation
  const localReceipt = await localDispatcher.dispatch(envelope);
  console.log('Simulated Receipt Status:', localReceipt.status); // 'accepted_pending'
  console.log('Simulated Delta Hash:', localReceipt.deltaHash);

  // Verify that the optimistic quad was loaded into the VKG client
  const allQuads = await vkgClient.getQuads();
  console.log('VKG Current Quads Count:', allQuads.length);

  // 6. Simulate synchronizing outbox to remote authority
  console.log('\n--- Step 2: Syncing outbox to remote authority ---');
  
  // Set execution environment to online state
  setNetworkOffline(false);
  setCurrentPrincipal({ id: 'usr-member-99', role: 'member' });

  try {
    await localDispatcher.syncOutbox(remoteDispatcher);
    console.log('Sync outbox operation completed successfully.');
    
    // Check Zustand store state
    const latestReceipt = useActorOpsStore.getState().latestReceipt;
    console.log('Zustand Logged Receipt Status:', latestReceipt?.status); // 'applied_remote'
  } catch (error: any) {
    console.error('Sync failed:', error.message);
  }
}

runActorWorkflow().catch(console.error);
```
