# Process Intelligence Substrate (`v2030`)

The `v2030` library is the core process intelligence, workflow auditing, and intervention generation substrate of the Truex ecosystem. It ensures that system actions, user trajectories, and agent decisions conform to formal constraints, auditing traces dynamically using process mining principles, validating trajectory state machines, and generating ontology-aligned app interventions.

---

## 1. Title & Overview

### What the Library Does
The `v2030` process intelligence library provides runtime verification and auditing functions. It:
1. **Audits Process Conformance:** Inspects user/system execution traces against expected state graphs or workflow structures using process mining algorithms (e.g., Van der Aalst fitness metrics).
2. **Detects Process & Concept Drift:** Identifies statistical shifts in activity streams using sliding-window Jaccard distances smoothed via Exponentially Weighted Moving Averages (EWMA).
3. **Validates State Trajectories:** Enforces state progression constraints through formal transition systems (Transition Families), blocking transitions that bypass required intermediate states or execute out of sequence.
4. **Enforces Compliance and Safety Guards:** Rejects event logs violating safety rules (e.g., stopping execution chains for an actor after a command rejection).
5. **Generates App Interventions:** Projects diagnostic outcomes into ontology-aligned interventions (such as `PROMPT`, `REORDER`, `RECOMMEND`, `REMIND`, `SUPPRESS`, or `ESCALATE`) to guide user behavior and self-correct UI workflows.
6. **Stores Replay Lineage:** Captures execution artifacts locally or on the filesystem to enable auditability, deterministic verification, and offline replays.

### Connection to the Truex Platform
Within the Truex platform, trust is maintained through cryptographic truthfulness. Rather than trusting that database updates or UI layouts are correct, the platform uses `v2030` to continuously audit logs, check state progression against lawful boundaries, and produce cryptographic receipts. This ensures that every action taken across the client app, decentralized agents, and cloud sync layers is fully verifiable and mathematically sound.

## 2. Architectural & Philosophical Mapping

### Mappings to the Truex Execution Pillars
The `v2030` library aligns directly with the four core architectural pillars of the Truex substrate:

* **Intake:** The library exposes capabilities defined by structured `InputContract` schemas. Telemetry traces, user check-in events, and agent commands are treated as raw inputs that are parsed, validated, and normalized before execution.
* **Membrane:** Capability execution is encapsulated. Capabilities do not run arbitrary side effects; they execute as pure-functional processors within runtime boundaries. Furthermore, the local/mobile cache of replay logs acts as an offline-first storage membrane, preventing private interaction telemetry from leaking outside of authorized local storage.
* **Projection:** Audited outcomes are projected into state changes. Specifically, outputs are translated into `AppIntervention` objects defined using semantic RDF quads. These interventions dynamically adapt client application layouts, reorder room displays, prompt action sequences, or suppress notifications.
* **Supervision:** The `IntelligenceRunner` acts as a runtime supervisor. It records execution logs, validates input contracts, computes cryptographic delta hashes, and writes audit receipts to the SQLite database. It works alongside temporal guards like the `ComplianceSafetyGuard` to supervise and halt non-compliant operations.

### Mathematical Mapping to the Receipted Chatman Equation
The process intelligence substrate operates in strict conformance with the Receipted Chatman Equation:

$$R \vdash A = \mu(O^*)$$

Where:
* **$O^*$ (Lawful Closure Ontology):** The set of inputs, declared workflows, allowed state sequences (`TransitionFamily` schemas like `SermonFlow` or `OrderFlow`), and capability registrations defining valid states and operations.
* **$\mu$ (Transformation/Manufacturing Function):** The capability execution pipeline (`IntelligenceRunner.run` and individual capability `.run()` algorithms) which evaluates inputs against rules and models.
* **$A$ (Emitted Consequence):** The resulting process metrics (fitness scores, drift alerts) or structured `AppIntervention` actions (e.g., `PROMPT`, `SUPPRESS`, `REORDER`) designed to adapt system behavior.
* **$R$ (Receipt Lineage):** The `IntelligenceReceipt` containing execution timestamps, status flags, and a cryptographic `deltaHash` stored in the `actorReceipts` table of the local SQLite database.

## 4. API Contracts

### Data Interfaces

#### `InputContract`
Declares the properties and requirements of a capability's input.
```typescript
export interface InputContract {
  properties: Record<string, { type: string; description: string; required?: boolean }>;
}
```

#### `OutputContract`
Declares the properties returned by the capability's execution.
```typescript
export interface OutputContract {
  properties: Record<string, { type: string; description: string }>;
}
```

#### `IntelligenceReceipt`
The cryptographic audit record generated after executing a capability.
```typescript
export interface IntelligenceReceipt {
  id: string;
  capabilityId: string;
  timestamp: string;
  success: boolean;
  deltaHash: string;
  logs: string[];
  error?: string;
}
```

#### `ReplayArtifact`
Captures inputs, outputs, and logs for deterministic process replays.
```typescript
export interface ReplayArtifact {
  receiptId: string;
  capabilityId: string;
  timestamp: string;
  input: any;
  output: any;
  logs: string[];
}
```

#### `IntelligenceCapability`
The interface implemented by all process intelligence capabilities.
```typescript
export interface IntelligenceCapability {
  id: string;
  name: string;
  description: string;
  inputContract: InputContract;
  outputContract: OutputContract;
  run(input: any): Promise<{ success: boolean; result: any; logs: string[]; error?: string }>;
}
```

#### `AppIntervention`
The interface representing a semantic UX or system adaptation.
```typescript
export interface AppIntervention {
  id: string; // e.g. "urn:zoe:intervention:<uuid>"
  verb: 'PROMPT' | 'REORDER' | 'RECOMMEND' | 'REMIND' | 'SUPPRESS' | 'ESCALATE' | 'ASSIGN' | 'RECEIPT';
  rdfQuads: {
    subject: string;
    predicate: string;
    object: string;
    graph?: string;
  }[];
}
```

#### `TransitionFamily`
The interface representing a permitted state machine trajectory.
```typescript
export interface TransitionFamily {
  id: string;
  states: string[];
  allowedTransitions: Record<string, string[]>;
  initialState: string;
}
```

### Core Classes & Methods

#### `IntelligenceRunner`
The main executor for registered capabilities.
* **`static async run(capabilityId: string, input: any): Promise<IntelligenceReceipt>`**
  Validates input contract schemas, executes the capability, computes the cryptographic receipt hash, writes the replay file, logs the receipt in SQLite, and returns the receipt.
* **`static getReplayArtifact(receiptId: string): ReplayArtifact | null`**
  Retrieves a replay log from either the memory cache or filesystem using the receipt ID.
* **`static listReplays(): ReplayArtifact[]`**
  Lists all currently cached and filesytem-persisted replay logs.

#### `IntelligenceReceiptManager`
Verifies and lists the database records of executed processes.
* **`static async verify(receiptId: string): Promise<{ verified: boolean; receipt: any; error?: string }>`**
  Looks up a receipt by ID in the SQLite database and verifies its status.
* **`static async list(): Promise<any[]>`**
  Returns all logged receipts whose `actorRef` belongs to a process `IntelligenceCapability`.

#### `validateTrajectory`
* **`validateTrajectory(familyId: string, trajectory: string[]): { success: boolean; error?: string }`**
  Ensures that the input state sequence conforms to the specified `TransitionFamily` (e.g. `SermonFlow`, `OrderFlow`, `VolunteerFlow`), returning detailed skipped-state errors if any intermediate transitions were bypassed.

## 5. Usage Guide

Here is a production-ready, complete example showing how to run process audits, validate trajectories, and access cached replay logs:

```typescript
import { IntelligenceRunner } from './src/lib/v2030/intelligence/runner';
import { validateTrajectory } from './src/lib/v2030/intelligence/trajectories';
import { IntelligenceReceiptManager } from './src/lib/v2030/intelligence/receipts';

async function runProcessIntelligencePipeline() {
  console.log('--- 1. Enforcing Trajectory Gating ---');
  
  // Validating a conforming transition trajectory
  const validOrderPath = ['idle', 'cart_updated', 'address_added', 'processing', 'paid'];
  const validResult = validateTrajectory('OrderFlow', validOrderPath);
  console.log(`Conforming path valid: ${validResult.success}`); // Expected: true

  // Validating a trajectory that skips steps
  const skippedOrderPath = ['idle', 'processing', 'paid'];
  const invalidResult = validateTrajectory('OrderFlow', skippedOrderPath);
  console.log(`Bypassed path valid: ${invalidResult.success}`); // Expected: false
  if (!invalidResult.success) {
    console.error(`Gating Rejection: ${invalidResult.error}`);
    // Output: "TransitionFamilyRefused: Illegal transition from 'idle' to 'processing' in OrderFlow. Missing required intermediate states: cart_updated, address_added"
  }

  console.log('\n--- 2. Executing Process Conformance Audit ---');
  
  const auditInput = {
    declaredWorkflow: ['PublishSermon', 'SendNotification', 'VerifyFeed', 'ArchiveRecord'],
    actualEvents: ['PublishSermon', 'SendNotification', 'VerifyFeed', 'UpdateDatabase', 'ArchiveRecord']
  };

  // Run the capability through the central orchestrator runner
  const receipt = await IntelligenceRunner.run('jtbd-conformance-auditor', auditInput);
  console.log(`Receipt generated: ${receipt.id}`);
  console.log(`Delta Hash: ${receipt.deltaHash}`);
  console.log(`Process Audit Success: ${receipt.success}`);
  
  // View execution logs
  console.log('Audit Logs:');
  receipt.logs.forEach(log => console.log(`  ${log}`));

  console.log('\n--- 3. Verifying Receipt Lineage ---');
  
  // Verify receipt integrity
  const verification = await IntelligenceReceiptManager.verify(receipt.id);
  console.log(`Cryptographic verification status: ${verification.verified}`);

  console.log('\n--- 4. Replaying Execution Output ---');
  
  // Fetch replay artifact for diagnostics
  const artifact = IntelligenceRunner.getReplayArtifact(receipt.id);
  if (artifact) {
    console.log('Replay Output:', JSON.stringify(artifact.output, null, 2));
    /*
      Expected Output:
      {
        "fitness": 0.75,
        "precision": 0.75,
        "simplicity": 0.5,
        "verdict": "VARIANCE"
      }
    */
  }
}

// Execute the workflow
runProcessIntelligencePipeline().catch(console.error);
```
