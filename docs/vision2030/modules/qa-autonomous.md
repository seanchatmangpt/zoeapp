# Autonomous QA & Self-Healing Module (qa-autonomous)

## 1. Overview

The `qa-autonomous` module is a foundational pillar of the **Zoe 2030 Innovation Peak**, delivering continuous runtime state auditing, automatic test generation, and autonomous self-healing capabilities. Instead of relying solely on offline static testing or passive error logging, this module establishes an active closed-loop system directly within the runtime context.

### Core Capabilities

- **State Monitoring**: Periodically evaluates the running state of the application against business and system invariants.
- **Autonomous Self-Healing**: Detects state deviations and automatically applies healing/recovery actions to restore state sanity.
- **Dynamic Test Generation**: Automatically generates reproduction and regression tests tailored to specific state variances on the fly.
- **Local Test Execution**: Runs generated assertions in a sandboxed, low-overhead runner, creating complete execution logs (Receipt Lineage) confirming system correction.

## 2. Architectural & Philosophical Mapping

The `qa-autonomous` module is built in strict compliance with the **Receipted Chatman Equation**:

$$R \vdash A = \mu(O^*)$$

Where:

- **$O^*$ (Lawful Closure Ontology)**: The set of admissible operational state constraints. In this module, the $O^*$ is represented by the **Invariant Conditions** defined via the `InvariantChecker`. If state stays within $O^*$, the system is safe.
- **$\mu$ (Manufacturing/Transformation Function)**: The self-healing transitions executed by the `AutonomousRepairAgent`. It translates the detected state variance back to the expected state values.
- **$A$ (Emitted Consequence)**: The corrected, verified runtime state ($S'$) following the repair.
- **$R$ (Receipt Lineage)**: The cryptographic/execution proof confirming that the state transition was executed safely and validated successfully. This is generated by the `TestRunner` as a `TestResult` containing status and comprehensive execution logs.

### Philosophical Pipeline

```mermaid

    S[Active System State] -->|1. Polled by StateGetter| SM[StateMonitor]
    SM -->|2. Audits against Invariants| O["Lawful Closure Ontology (O*)"]
    O -->|Variance Detected| SV[StateVariance]
    SV -->|3. Triggers Repair| RA[AutonomousRepairAgent]
    RA -->|4. Invokes Dynamic Gen| TG[TestGenerator]
    TG -->|Generates Assertion| T[Dynamic Reproduction Test]
    RA -->|5. Transforms State (μ)| S_Prime[Restored State (A)]
    RA -->|6. Asserts Restored State| TR[TestRunner]
    TR -->|Runs assertion on S_Prime| T
    TR -->|7. Emits Log / Verification| R[Receipt Lineage (R)]
```

## 3. Source Code Structure

Located at `src/framework/2030/qa-autonomous`, the module exhibits a decoupled, single-responsibility architecture:

```
src/framework/2030/qa-autonomous/
├── index.ts                  # Entrypoint exporting all interfaces, classes, and types
├── types.ts                  # Shared data models, contracts, and callback signatures
├── StateMonitor.ts           # Periodic state checking and invariant validation
├── TestGenerator.ts          # Dynamic test template generator for caught variances
├── TestRunner.ts             # Sandboxed test execution and log-aggregator
├── AutonomousRepairAgent.ts  # Core coordinator stitching monitor, runner, and healing
└── __tests__/
    └── AutonomousRepairAgent.test.ts # Exhaustive unit and integration test suite
```

### File Responsibilities

- **`index.ts`**: Provides clean, consolidated exports to ensure third-party modules can access the full API surface under a unified namespace.
- **`types.ts`**: Defines the strict TypeScript structures that govern data passing between components, ensuring structural parity and interface safety.
- **`StateMonitor.ts`**: Manages the runtime polling loop. It decouples the state acquisition logic (`StateGetter`) from the invariant checks, enabling hot-reloading of validation logic.
- **`TestGenerator.ts`**: Translates a structured `StateVariance` object into executable assertion code and documentation of the reproduction steps.
- **`TestRunner.ts`**: Manages runtime validation execution, catching all synchronous or asynchronous failures and translating them into structured, readable test results.
- **`AutonomousRepairAgent.ts`**: The main orchestrator that coordinates the lifecycle. It receives variance events, applies state mutations, generates tests, runs the verification checks, and triggers external listener callbacks.

## 4. Public Interfaces / API Contracts

### Data Models & Types (`types.ts`)

#### `StateVariance`

Represents an observed deviation from the Lawful Closure Ontology.

```typescript
export interface StateVariance {
  key: string; // The state property key that deviated
  expected: any; // The expected compliant value
  actual: any; // The actual out-of-bounds value
  timestamp: number; // Epoch timestamp of the detection
  severity: 'low' | 'medium' | 'high'; // Severity rating of the violation
}
```

#### `TestResult`

Represents the outcome of a verification run.

```typescript
export interface TestResult {
  success: boolean; // True if the test passed
  error?: string; // The error message if the assertion failed or threw
  logs: string[]; // Sequential execution logs recorded during the test
}
```

#### `RepairPlan`

Associates a variance with a concrete repair routine.

```typescript
export interface RepairPlan {
  variance: StateVariance; // The target variance
  action: () => Promise<void>; // The async routine to heal the state
  description: string; // Descriptive overview of the repair strategy
}
```

#### `AutonomousConfig`

Configures the operational behavior of the repair agent.

```typescript
export interface AutonomousConfig {
  monitorIntervalMs: number; // Frequency of invariant polling
  autoRepair: boolean; // Whether to automatically repair variances
  onVarianceDetected?: (variance: StateVariance) => void; // Triggered upon variance discovery
  onRepairCompleted?: (result: TestResult) => void; // Triggered after repair/test run
}
```

#### Functional Signatures

```typescript
export type StateGetter = () => Record<string, any>;
export type StateSetter = (state: Record<string, any>) => void;
export type InvariantChecker = (state: Record<string, any>) => StateVariance[];
```

## 5. Usage Guide

Here is a comprehensive, production-ready example demonstrating how to integrate `qa-autonomous` into an application store (representing system state), define invariants, monitor drift, and receive self-healing confirmations.

```typescript
import {
  AutonomousRepairAgent,
  StateVariance,
  TestResult,
} from '@/src/framework/2030/qa-autonomous';

// 1. Define application state structure and simple store
interface AppState {
  systemMode: 'active' | 'maintenance' | 'corrupt';
  userSessionCount: number;
  databaseConnected: boolean;
}

let appState: AppState = {
  systemMode: 'active',
  userSessionCount: 0,
  databaseConnected: true,
};

const getState = (): Record<string, any> => appState;
const setState = (nextState: Record<string, any>): void => {
  appState = nextState as AppState;
  console.log('[System State Updated]:', appState);
};

// 2. Formulate Invariant Rules (O* Lawful Closure Ontology)
const checkInvariants = (state: Record<string, any>): StateVariance[] => {
  const variances: StateVariance[] = [];

  // Invariant A: If database is disconnected, status should NOT remain 'active'
  if (!state.databaseConnected && state.systemMode === 'active') {
    variances.push({
      key: 'systemMode',
      expected: 'maintenance',
      actual: state.systemMode,
      timestamp: Date.now(),
      severity: 'high',
    });
  }

  // Invariant B: Session counts must never drop below 0
  if (state.userSessionCount < 0) {
    variances.push({
      key: 'userSessionCount',
      expected: 0,
      actual: state.userSessionCount,
      timestamp: Date.now(),
      severity: 'medium',
    });
  }

  return variances;
};

// 3. Configure the Autonomous Repair Agent
const agent = new AutonomousRepairAgent(getState, setState, checkInvariants, {
  monitorIntervalMs: 1000, // Audit state every 1 second
  autoRepair: true, // Enable self-healing
  onVarianceDetected: (variance: StateVariance) => {
    console.warn(
      `[ALERT] Variance detected in key "${variance.key}". ` +
        `Expected: ${variance.expected}, Got: ${variance.actual} (Severity: ${variance.severity})`
    );
  },
  onRepairCompleted: (result: TestResult) => {
    console.log('[HEALING COMPLETED] Verification Result:', result.success ? 'PASSED' : 'FAILED');
    console.log('[HEALING LOGS]:\n', result.logs.join('\n'));
  },
});

// 4. Start the Agent
agent.start();
console.log('Autonomous State Monitoring and Self-Healing active.');

// 5. Simulate a state corruption event (Out of Spec)
setTimeout(() => {
  console.log('\n--- Simulating State Corruption (Database Dropped but System Mode Active) ---');
  setState({
    ...appState,
    databaseConnected: false,
    systemMode: 'active', // Violates Invariant A
  });
}, 2000);

// 6. Stop agent on exit/cleanup
setTimeout(() => {
  agent.stop();
  console.log('\nMonitoring stopped. Demo terminated.');
}, 5000);
```
