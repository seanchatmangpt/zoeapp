# Zoe 2030 Authentication & Identity Gating Layer

The `auth` module (located under `src/framework/auth`) provides the decoupled authentication, authorization, role-based access control (RBAC), multi-factor verification, zero-knowledge proofs (ZKP), and continuous behavioral biometric telemetry for the **Zoe 2030 Innovation Peak** framework. Decoupled from specific routing or UI packages, it serves as the foundational trust membrane of the client substrate.

---

## 1. Overview

In high-assurance enterprise and agentic environments, authentication cannot remain a simple binary session check. The Zoe 2030 framework addresses these challenges by introducing a multi-dimensional security membrane.

The `auth` module addresses several core needs:

1. **Decoupled Route Gating:** Pure, deterministic gate-checking logic separate from visual libraries, allowing the same rules to run in React Native, Web, or background agent runtimes.
2. **Context-Aware Identity Management:** A hierarchical multi-tenant identity manager that handles tenant switching, group scope switching, and inherited role-permission evaluation.
3. **Multi-Factor Verification (MFA) Abstraction:** A strategy-agnostic MFA controller supporting TOTP, SMS, Email, and Biometrics, including session-based grace periods and functional wrappers.
4. **Privacy-Preserving Assertions (ZKP):** Zero-Knowledge Proof validation that permits identity claims (e.g., age thresholds, role membership, country of origin) to be verified without disclosing the underlying user values.
5. **Continuous Behavioral Biometrics:** Passive telemetry monitoring typing speeds (inter-keystroke intervals) and navigation rhythms to calculate a real-time trust score, warning the system if a session has been hijacked by a script, macro, or bot.

## 4. API Contracts

### 4.1 Core Gating Layer (`types.ts` & `guards.ts`)

#### `IdentityBoundary`

```typescript
type IdentityBoundary = string;
```

Represents trust boundaries. The default hierarchy configuration is:
`['anonymous', 'authenticated', 'verified', 'mfa_verified']`

#### `Disclosure`

```typescript
type Disclosure = string;
```

Consents or disclosures required from the user before admitting them (e.g., `'terms_accepted'`).

#### `RefusalReason`

A detailed payload emitted when access is denied:

- `code: string`: Error code identifier (e.g., `'UNAUTHENTICATED'`, `'MISSING_ROLE'`).
- `message: string`: Human-readable refusal description.
- `requiredIdentityBoundary?: IdentityBoundary`: The boundary required by the gate.
- `actualIdentityBoundary?: IdentityBoundary`: The active boundary of the participant.
- `missingDisclosures?: readonly Disclosure[]`: List of disclosures the participant has not consented to.
- `missingRoles?: readonly Role[]`: List of roles missing from the participant.
- `missingPermissions?: readonly Permission[]`: List of permissions missing from the participant.
- `[key: string]: any`: Extensible metadata.

#### `ParticipantBasis`

The minimal data model representing the participant being evaluated:

- `identityBoundary: IdentityBoundary`: Active security level.
- `disclosures: readonly Disclosure[]`: Consolidated set of completed disclosures.
- `roles?: readonly Role[]`: Roles assigned to the participant.
- `permissions?: readonly Permission[]`: Direct permissions assigned.

#### `RouteDefinition`

The gating requirements defined for a route:

- `requiredIdentityBoundary?: IdentityBoundary` (Optional): Required security boundary level.
- `requiredDisclosures?: readonly Disclosure[]` (Optional): Required disclosures.
- `requiredRoles?: readonly Role[]` (Optional): Required roles.
- `requiredPermissions?: readonly Permission[]` (Optional): Required permissions.
- `customGuard?: (participant: ParticipantBasis) => RefusalReason | null` (Optional): Custom validation function.

#### `admitRoute`

```typescript
function admitRoute(
  participant: ParticipantBasis | null | undefined,
  route: RouteDefinition,
  hierarchy?: readonly IdentityBoundary[]
): AdmitRouteResult;
```

- **Description:** A pure checking function that determines whether a participant meets a route's requirements.
- **Parameters:**
  - `participant`: The participant data to check. If null/undefined, defaults to an `'anonymous'` boundary with no disclosures.
  - `route`: Gating requirements.
  - `hierarchy`: Custom identity level boundary hierarchy. Defaults to `DEFAULT_IDENTITY_HIERARCHY`.
- **Returns:** `AdmitRouteResult` containing `admitted: boolean` and an optional `refusal: RefusalReason`.

### 4.2 React Providers & Context Hooks

#### `AuthProvider`

React context provider component.

- **Props (`AuthProviderProps`):**
  - `children: ReactNode`: Children to render.
  - `getInitialSession?: () => Promise<TUser | null>`: Async function to retrieve the starting session on mount.
  - `onAuthStateChange?: (callback: (event: string, session: TUser | null) => void) => () => void`: Event subscription callback for external auth changes (e.g. Supabase/Firebase listener).
  - `transitionDurationMs?: number`: Time in milliseconds to show transitional signin/signout screens. Defaults to `850` ms.
  - `resolveParticipant?: (session: TUser) => ParticipantBasis`: Optional mapper function to convert session model into a `ParticipantBasis` structure.
- **Context Value (`AuthContextValue`):**
  - `session: TUser | null`: Active session user object.
  - `participant: ParticipantBasis | null`: Active resolved participant attributes.
  - `loading: boolean`: True during initial session loading.
  - `isTransitioning: boolean`: True during signin/signout transition periods.
  - `transitionType: 'signin' | 'signout' | null`: Specifies the current transition type.
  - `setIsTransitioning: (val: boolean) => void`: Manually triggers transition overlays.
  - `setSession: (session: TUser | null) => void`: Updates active session, auto-triggering signin/signout transitions.

#### `ProtectedRoute`

Decoupled visual component used to gate React subtrees.

- **Props (`ProtectedRouteProps`):**
  - `route: RouteDefinition`: Gating requirements.
  - `children: React.ReactNode`: Content to render if admitted.
  - `resolveParticipant: (session: any) => ParticipantBasis`: Callback to convert session object into participant.
  - `loadingComponent?: React.ReactNode`: Component to render while session resolves.
  - `fallback?: React.ReactNode | ((refusal: RefusalReason) => React.ReactNode)`: Render-prop or node rendered if access is denied.
  - `verifyExternalState?: (route: RouteDefinition) => Promise<{ verified: boolean; refusal?: RefusalReason }>`: Async external check (e.g., verifying a receipt payload or deep DB checks).
  - `externalCheckingComponent?: React.ReactNode`: Component to show while external checks run.
  - `hierarchy?: readonly IdentityBoundary[]`: Custom security level hierarchy.
  - `participant?: ParticipantBasis`: Optional explicit override of user state.

#### `useRBAC`

React hook to check roles and permissions.

- **Returns:**
  - `hasRole: (role: Role) => boolean`
  - `hasPermission: (permission: Permission) => boolean`
  - `hasAnyRole: (roles: readonly Role[]) => boolean`
  - `hasAllRoles: (roles: readonly Role[]) => boolean`
  - `hasAnyPermission: (permissions: readonly Permission[]) => boolean`
  - `hasAllPermissions: (permissions: readonly Permission[]) => boolean`
  - `roles: readonly Role[]`
  - `permissions: readonly Permission[]`
  - `loading: boolean`

### 4.3 Multi-Tenant Identity Management (`identity/IdentityManager.ts`)

#### `IdentityManager`

Fluent API class that tracks organizational structures and membership contexts.

- **Constructor:**
  ```typescript
  constructor(
    initialState: IdentityState,
    tenants?: Tenant[],
    groups?: Group[],
    roles?: Role[]
  )
  ```
- **Methods:**
  - `switchTenant(tenantId: TenantId): this`: Changes active tenant. Clears active group if the active group belongs to another tenant. Throws if the user does not have membership in the target tenant.
  - `switchGroup(groupId: GroupId): this`: Switches active group context. Validates that the group belongs to the active tenant and that the user is a member of the group (or a parent group).
  - `getContext(): IdentityContext`: Resolves user roles, groups, and permissions. Calculates inherited memberships (permissions from parent groups are automatically inherited by descendants).
  - `admitToGroup(groupId: GroupId, roleIds: RoleId[]): this`: Simulates assigning group memberships dynamically.
  - `getState(): Readonly<IdentityState>`: Returns the active user state.

### 4.4 Multi-Factor Authentication (`mfa/`)

#### `MfaProvider`

- **Props (`MfaProviderProps`):**
  - `onInitiateChallenge: (strategy: MfaStrategy) => Promise<MfaChallenge>`: Callback to trigger challenge generation on the server.
  - `onVerifyCode: (challengeId: string, code: string) => Promise<{ success: boolean; token?: string }>`: Callback to verify code.
  - `verificationGracePeriod?: number`: Duration in milliseconds before MFA needs re-verification.
- **Context Value (`MfaContextValue`):**
  - `isVerified: boolean`: True if MFA verified.
  - `lastVerifiedAt?: number`: Millisecond timestamp of verification.
  - `activeChallenge?: MfaChallenge | null`: The currently unresolved challenge.
  - `verify: (options?: MfaVerificationOptions) => Promise<MfaVerificationResult>`: Initiates MFA check. Returns a promise resolving when verified/cancelled.
  - `confirm: (code: string) => Promise<boolean>`: Confirms the active challenge code.
  - `cancel: () => void`: Cancels the active challenge.

#### `useMfaVerification`

Public Hook.

- **Methods/Properties:**
  - `verify: (options?: MfaVerificationOptions) => Promise<MfaVerificationResult>`
  - `confirm: (code: string) => Promise<boolean>`
  - `cancel: () => void`
  - `withMfa: <T>(action: () => Promise<T> | T, options?: MfaVerificationOptions) => Promise<T | null>`: Executes `action` only if MFA passes. Returns `null` otherwise.
  - `activeChallenge: MfaChallenge | null`
  - `isPending: boolean`
  - `isVerified: boolean`
  - `lastVerifiedAt?: number`

### 4.5 Zero-Knowledge Proofs (`zkp/`)

#### `ZkEngine` (Singleton instance `zkEngine`)

Implements `ZkProvider`.

- **Methods:**
  - `verify(claim: ZkClaim, proof: ZkProof): Promise<ZkVerificationResult>`: Verifies a ZKP assertion. Ensures claim and proof identifiers match, then runs cryptographic structure checks.
  - `performCryptographicVerification(claim: ZkClaim, proof: ZkProof): Promise<boolean>` (Private): Structural integrity validations. Can be wired to WASM-based cryptographic engines (such as `@truex/zkp` or SnarkJS).

#### `useZkClaimVerifier`

State hook for UI proof verification workflows.

- **Returns:**
  - `verifyClaim: (claim: ZkClaim, proof: ZkProof) => Promise<ZkVerificationResult>`
  - `isVerifying: boolean`
  - `result: ZkVerificationResult | null`
  - `error: string | null`
  - `reset: () => void`

### 4.6 Continuous Behavioral Biometrics (`behavioral/`)

#### `useBehavioralAuth`

React hook that passive-samples user interactions.

- **Parameters (`BehavioralAuthOptions`):**
  - `updateInterval?: number`: Intervals in milliseconds between trust score recalculations. Defaults to `5000` ms.
  - `sensitivity?: number`: Factor multiplying score reductions (volatility indicator). Defaults to `0.5`.
- **Returns (`BehavioralAuthState`):**
  - `trustScore: number`: Session confidence indicator between `0.0` (compromised) and `1.0` (validated).
  - `metrics: BehavioralMetrics`: Current parameters (average typing speed interval, simulated touch pressure, navigation actions per minute).
  - `isActive: boolean`: Active status of listeners.
  - `recordKeystroke: () => void`: Manual keystroke recorder (useful for mobile UI platforms where window key listeners are blocked).
  - `recordInteraction: () => void`: Manual navigation/interaction event recorder.

```tsx
      {activeTab === 'main' && (
        <View style={styles.card}>
          <Text style={styles.cardTitle}>Protected Routing Test</Text>
          <Text style={styles.infoText}>
            Current Boundary: {session ? 'verified' : 'anonymous'}
          </Text>

          <ProtectedRoute
            route={adminPanelGating}
            resolveParticipant={resolveParticipantFromSession}
            loadingComponent={<ActivityIndicator size="small" />}
            fallback={(refusal) => (
              <View style={styles.refusalBanner}>
                <Text style={styles.refusalText}>
                  Access Denied: {refusal.message} ({refusal.code})
                </Text>
                <Text style={styles.refusalSub}>
                  Required identity boundary level: "{refusal.requiredIdentityBoundary}"
                </Text>
              </View>
            )}>
            <View style={styles.adminSuccess}>
              <Text style={styles.successText}>
                ✔ ADMITTED: Secret Admin Panel (Lawful Closure Confirmed)
              </Text>
              <Text style={styles.infoText}>
                Available Roles: {activeContext.roles.map((r) => r.name).join(', ')}
              </Text>
            </View>
          </ProtectedRoute>

          <TouchableOpacity style={[styles.button, { marginTop: 15 }]} onPress={handleForceLogOut}>
            <Text style={styles.buttonText}>Simulate Session Logout</Text>
          </TouchableOpacity>
        </View>
      )}

      {/* Identity Tab */}
      {activeTab === 'identity' && (
        <View style={styles.card}>
          <Text style={styles.cardTitle}>IdentityManager Hierarchy Switches</Text>
          <Text style={styles.infoText}>Active Tenant: {activeContext.tenant?.name || 'None'}</Text>
          <Text style={styles.infoText}>Active Group: {activeContext.group?.name || 'None'}</Text>

          <View style={styles.buttonRow}>
            {mockTenants.map((t) => (
              <TouchableOpacity
                key={t.id}
                onPress={() => handleSwitchTenant(t.id)}
                style={[
                  styles.buttonMini,
                  tenantState.activeTenantId === t.id && styles.buttonMiniActive,
                ]}>
                <Text style={styles.buttonTextMini}>{t.name}</Text>
              </TouchableOpacity>
            ))}
          </View>

          <Text style={styles.sectionLabel}>Switch Group:</Text>
          <View style={styles.buttonRow}>
            {mockGroups
              .filter((g) => g.tenantId === tenantState.activeTenantId)
              .map((g) => (
                <TouchableOpacity
                  key={g.id}
                  onPress={() => handleSwitchGroup(g.id)}
                  style={[
                    styles.buttonMini,
                    tenantState.activeGroupId === g.id && styles.buttonMiniActive,
                  ]}>
                  <Text style={styles.buttonTextMini}>{g.name}</Text>
                </TouchableOpacity>
              ))}
          </View>
        </View>
      )}

      {/* MFA Tab */}
      {activeTab === 'mfa' && (
        <View style={styles.card}>
          <Text style={styles.cardTitle}>Multi-Factor Verification</Text>
          <Text style={styles.infoText}>MFA Verified: {mfa.isVerified ? 'Yes' : 'No'}</Text>
          {mfa.lastVerifiedAt && (
            <Text style={styles.infoText}>
              Last Verified: {new Date(mfa.lastVerifiedAt).toLocaleTimeString()}
            </Text>
          )}

          {mfa.isPending && mfa.activeChallenge && (
            <View style={styles.challengeBox}>
              <Text style={styles.challengeText}>
                Challenge Initiated: Enter code for {mfa.activeChallenge.strategy.toUpperCase()}
              </Text>
              <Text style={styles.challengeSub}>Code is '424242' for simulator checks</Text>
              <TextInput
                style={styles.input}
                value={mfaCode}
                onChangeText={setMfaCode}
                placeholder="Enter 6-digit Code"
                keyboardType="numeric"
              />
              <View style={styles.buttonRow}>
                <TouchableOpacity
                  style={styles.buttonMini}
                  onPress={async () => {
                    const verified = await mfa.confirm(mfaCode);
                    if (verified) setMfaCode('');
                  }}>
                  <Text style={styles.buttonTextMini}>Verify Code</Text>
                </TouchableOpacity>
                <TouchableOpacity
                  style={[styles.buttonMini, { backgroundColor: '#7f8c8d' }]}
                  onPress={() => mfa.cancel()}>
                  <Text style={styles.buttonTextMini}>Cancel</Text>
                </TouchableOpacity>
              </View>
            </View>
          )}

          {!mfa.isPending && (
            <TouchableOpacity style={styles.button} onPress={handleSensitiveWriteAction}>
              <Text style={styles.buttonText}>Execute Secure Ledger Write</Text>
            </TouchableOpacity>
          )}
        </View>
      )}

      {/* ZKP Tab */}
      {activeTab === 'zkp' && (
        <View style={styles.card}>
          <Text style={styles.cardTitle}>Zero-Knowledge Proof Assertion</Text>
          <Text style={styles.infoText}>
            Assert age thresholds privately without leaking date of birth.
          </Text>

          {zkp.isVerifying ? (
            <ActivityIndicator size="small" color="#3498db" />
          ) : (
            <TouchableOpacity style={styles.button} onPress={handleExecuteZkProof}>
              <Text style={styles.buttonText}>Generate & Verify Age Proof</Text>
            </TouchableOpacity>
          )}

          {zkpOutput ? (
            <View style={styles.zkpOutputCard}>
              <Text style={styles.zkpOutputText}>{zkpOutput}</Text>
            </View>
          ) : null}
        </View>
      )}
    </ScrollView>
  );
}
```

## 6. Testing

The reliability, security, and gating boundaries of the `auth` module are backed by a complete test suite spanning core logic, visual components, context providers, and sub-domains.

### 6.1 Unit & Integration Test Targets

1. **Route Admission Logic (`guards.test.ts`):** Verifies that identity levels, required permissions, roles, custom guards, and disclosure agreements are parsed correctly, admitting or refusing participants with appropriate refusal codes.
2. **UI Route Gating (`ProtectedRoute.test.tsx`):** Renders protected subtrees inside Mock AuthProviders, validating that loading templates, fallback error templates, and verifyExternalState async operations trigger successfully.
3. **Session Context Management (`AuthProvider.test.tsx` & `hooks.test.tsx`):** Confirms that login/logout operations update the contexts correctly and trigger transition animations/delays.
4. **Continuous Biometric Telemetry (`useBehavioralAuth.test.ts`):** Mocks keyboard and touch events, validating typing intervals and navigation rhythms to verify that anomalies reduce the trust score under designated sensitivities.
5. **Multi-Factor Lifecycles (`mfa.test.tsx`):** Asserts challenge initiations, code validation inputs, grace period timers, and cancellation triggers.
6. **Hierarchical tenant RBAC context (`IdentityManager.test.ts`):** Validates tenant swapping constraints and confirms that permissions from parent groups inherit down to sub-groups.
7. **Zero-Knowledge Assertions (`zkp.test.ts`):** Verifies that mismatching proof IDs fail immediately and validates hook state transitions (isVerifying, result, error) on successful proof checks.

### 6.2 Running the Tests

To run the full suite of unit and integration tests specifically targeting the `auth` module, execute the following command in the workspace directory:

```bash
npm test src/framework/auth
```
